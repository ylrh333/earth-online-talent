# 地球 Online 职业 Mod 规范说明

## 1. 目标

职业 Mod 是一个具体岗位的可玩模拟包。

它不是职业百科，也不是普通问卷，而是让玩家进入一个真实岗位，在有限信息、时间压力、人际关系、资源约束和突发事件中做判断、沟通和行动。

一个合格的职业 Mod 应该回答三个问题：

- 这个岗位真实在处理什么问题？
- 玩家做出不同选择后，局面会怎么变化？
- 什么行为能说明玩家更像这个岗位里的高手？

职业 Mod 的真实性取决于作者提供的信息密度。AI 可以蒸馏和整理，但真实素材必须来自作者。

制作前必须尽量收集：

- 30 天职业流水账：按 Day 1 到 Day 30 记录真实工作日，每天尽量写清早上、下午、加班/突发、遇到谁、发生什么、怎么判断和行动、结果或遗留问题。
- 工作流程：岗位小白第一天、第一周会遇到什么，典型工作怎么从开始流转到结束。
- 常见问题：真实工作中的异常、返工、投诉、扯皮、信息缺口、时间压力。
- 人际关系：会和谁打交道，他们各自的目标、压力、话术和不愿承担的责任。
- 高手判断标准：老手会先看什么、先问什么、不会乱承诺什么。
- 新人翻车点：小白最容易误解什么、漏掉什么、被谁带节奏。
- 作者经历：脱敏后的真实案例、场景、术语、流程和边界。

## 2. 职业粒度

Mod 应该写到具体岗位，而不是只写行业大类。

推荐粒度：

- 汽车质量工程师
- 汽车工艺工程师
- 机械结构设计师
- 供应商质量工程师
- 设备维修工程师
- 生产班组长
- 模具工程师
- 项目工程师

不推荐粒度：

- 机械行业
- 制造业
- 汽车行业
- 工程师

原因：行业太大，玩家体验会变成泛泛介绍；岗位越具体，冲突、任务、术语和判断标准越真实。

## 3. 推荐目录结构

每个 Mod 使用一个独立目录，目录名使用小写英文和连字符。

```text
career-mods/
  automotive-quality-engineer/
    mod.json
    world.md
    player.md
    roles.json
    missions.json
    events.json
    scenes.json
    knowledge.md
    scoring.json
    system-prompt.md
    endings.md
```

## 4. 文件说明

### 4.1 `mod.json`

Mod 的基础信息。

```json
{
  "id": "automotive-quality-engineer",
  "title": "汽车质量工程师模拟器",
  "version": "0.1.0",
  "author": "你的名字或昵称",
  "industry": "汽车制造",
  "role": "质量工程师",
  "difficulty": "medium",
  "tags": ["汽车", "质量", "8D", "客户投诉", "量产"],
  "description": "玩家扮演汽车零部件供应商质量工程师，处理客户投诉、制程异常、来料问题和审核压力。",
  "estimatedMinutes": 30,
  "language": "zh-CN"
}
```

要求：

- `id` 必须全仓库唯一。
- `title` 要明确是哪个岗位。
- `difficulty` 可选：`easy`、`medium`、`hard`。
- `description` 不要写成宣传语，要说明玩家会体验什么。

### 4.2 `world.md`

岗位所在的真实环境。

建议包含：

- 公司类型
- 上下游关系
- 核心压力
- 常见矛盾
- 行业规则
- 不能违背的现实约束

示例：

```md
# 世界背景

你所在公司是一家汽车零部件供应商，主要给主机厂供货。公司已经进入量产阶段，客户对交付、质量和响应速度要求很高。

质量工程师的核心压力来自三方：

- 客户要求快速闭环和可验证措施；
- 生产部门担心停线、返工和效率下降；
- 老板关注成本、赔偿和客户关系。

玩家不能随意编造数据，不能跳过现场确认，不能把临时措施当成永久措施。
```

### 4.3 `player.md`

玩家身份设定。

建议包含：

- 玩家岗位
- 当前级别
- 职责范围
- 可调用资源
- 权限边界
- 初始状态

示例：

```md
# 玩家身份

你是汽车零部件供应商的一名质量工程师，入职 3 年，负责某量产项目的客户质量问题和过程质量跟踪。

你可以调用：

- 检验记录
- 不良样件
- 生产日报
- 工艺文件
- 供应商来料记录
- 现场操作员访谈

你不能直接决定停线、换供应商或对客户承诺赔偿，这些需要上级批准。
```

### 4.4 `roles.json`

NPC 角色列表。

```json
[
  {
    "id": "customer_sqe",
    "name": "客户 SQE",
    "role": "客户质量工程师",
    "personality": "直接、强势、重证据",
    "goal": "要求供应商快速定位原因并提交有效 8D",
    "knows": ["客户现场出现批量装配异常", "客户已经暂停部分批次使用"],
    "hiddenPressure": "内部正在考虑切换供应商"
  },
  {
    "id": "production_supervisor",
    "name": "生产主管",
    "role": "生产主管",
    "personality": "务实、怕停线、对质量追责敏感",
    "goal": "维持产量，不希望问题被扩大",
    "knows": ["最近换过一批操作员", "夜班返修率偏高"],
    "hiddenPressure": "本月产量指标已经落后"
  }
]
```

要求：

- 每个 NPC 都要有目标和压力。
- NPC 之间可以有利益冲突。
- 不要让所有 NPC 都配合玩家。

### 4.5 `missions.json`

主线任务。

```json
[
  {
    "id": "customer-complaint-8d",
    "title": "客户投诉：装配孔位偏差",
    "brief": "客户反馈某批零件装配困难，要求 24 小时内提交初步分析和围堵措施。",
    "startState": {
      "timeLimitHours": 24,
      "customerTrust": 60,
      "productionPressure": 75,
      "availableClues": ["客户照片", "出货批次", "检验记录"]
    },
    "objectives": [
      "确认问题范围",
      "提出临时围堵措施",
      "定位可能原因",
      "准备客户沟通口径"
    ],
    "successCriteria": [
      "没有跳过现场和数据确认",
      "区分临时措施和永久措施",
      "能说明证据链",
      "对客户承诺不过度"
    ],
    "failureRisks": [
      "盲目甩锅生产或供应商",
      "只写报告不看样件",
      "承诺无法验证的根因",
      "忽视库存和在途品围堵"
    ]
  }
]
```

要求：

- 每个任务必须有清晰冲突。
- 每个任务必须有成功标准和失败风险。
- 不要只写“完成某报告”，要写玩家需要判断什么。

### 4.6 `events.json`

随机事件或阶段事件。

```json
[
  {
    "id": "boss-asks-for-fast-answer",
    "trigger": "玩家调查超过 3 轮仍未给出客户回复",
    "title": "老板催你先给客户一个说法",
    "content": "老板说：客户那边已经很不耐烦了，你先写一个供应商来料异常的结论过去。",
    "pressure": ["时间压力", "上级压力", "证据不足"],
    "goodResponse": "玩家应说明目前证据不足，可以先提交临时围堵和调查计划，但不能虚构根因。",
    "badResponse": "玩家直接按老板要求写死根因，后续一旦证据不符会损害客户信任。"
  }
]
```

要求：

- 事件要制造真实压力。
- 事件不能只有正确答案，要允许玩家用不同方式处理。
- 事件应推动剧情，而不是单纯插入知识点。

### 4.7 `scenes.json`

网页游戏场景画面提示。

```json
[
  {
    "id": "customer-quality-war-room",
    "title": "客户投诉质量会议室",
    "location": "供应商质量办公室",
    "imagePrompt": "网页游戏写实背景图：汽车零部件供应商的质量办公室，一名质量工程师面对电脑、问题清单、检验记录和客户投诉照片，旁边有生产主管和质量经理在讨论，桌上有零件样件和红色标记的批次清单；画面真实、有工作压力，不出现真实公司 Logo、客户名称、车标或机密图纸。",
    "mood": "紧张、证据不足、客户催促",
    "visibleObjects": ["电脑", "问题清单", "检验记录", "零件样件", "批次清单"]
  }
]
```

要求：

- `imagePrompt` 用来生成或描述网页游戏背景图，必须服务于真实工作场景。
- 场景要包含具体工作物品、人物关系和问题压力。
- 不要写纯氛围图，不要出现真实公司 Logo、客户名称、车标、报价、图纸或内部文件。
- 如果玩家反馈图片不真实，优先修改这里。

### 4.8 `knowledge.md`

岗位知识库。

内容可以包括：

- 常用术语
- 常见流程
- 判断原则
- 典型工具
- 行业禁忌
- 真实工作中的灰度地带

示例：

```md
# 岗位知识

## 8D

8D 是客户质量问题中常见的闭环方法，但真实工作里不能把 8D 当模板填空。关键是证据链、围堵措施、根因验证和永久措施有效性。

## 5Why

5Why 不是机械地问五次为什么，而是沿着可验证的因果链继续追问。每一层原因都需要证据支持。

## 围堵

围堵对象通常包括客户现场品、在途品、成品库存、半成品、原材料和当前生产批次。
```

要求：

- 可以写行业术语，但必须解释。
- 不要堆百科，要写和游戏判断有关的知识。
- 真实从业经验优先于空泛理论。

### 4.9 `scoring.json`

评分维度和判定标准。

```json
{
  "dimensions": [
    {
      "key": "site-sense",
      "name": "现场感",
      "description": "是否会回到现场、样件、数据和过程，而不是只靠猜测。",
      "positiveSignals": [
        "主动查看不良样件",
        "核对批次、时间、工位和检验记录",
        "区分事实、假设和待验证信息"
      ],
      "negativeSignals": [
        "没有证据就下结论",
        "只听单方说法",
        "忽视在制品和库存风险"
      ]
    },
    {
      "key": "communication",
      "name": "沟通闭环",
      "description": "能否面对客户、老板和生产部门时保持清晰边界和推进节奏。",
      "positiveSignals": [
        "说明现状、风险、下一步和时间点",
        "对客户不过度承诺",
        "让相关部门明确责任和交付物"
      ],
      "negativeSignals": [
        "用情绪对抗客户",
        "对老板盲目服从",
        "只催别人但不给路径"
      ]
    }
  ],
  "endingLevels": [
    {
      "level": "excellent",
      "name": "像真实老手",
      "minScore": 85
    },
    {
      "level": "pass",
      "name": "能扛住基本工作",
      "minScore": 60
    },
    {
      "level": "risk",
      "name": "容易在真实岗位翻车",
      "minScore": 0
    }
  ]
}
```

要求：

- 每个维度必须有正向信号和负向信号。
- 评分维度不要太多，建议 4 到 8 个。
- 评分应评玩家行动，不要评玩家性格。

### 4.10 `system-prompt.md`

AI 裁判和 NPC 的运行规则。

建议包含：

- 游戏目标
- AI 扮演哪些角色
- 如何推进剧情
- 如何判断玩家行动
- 什么时候给提示
- 什么时候惩罚不合理行为
- 禁止行为

示例：

```md
# 系统提示词

你是「地球 Online」职业模拟器的游戏主持人、职业裁判和 NPC 扮演者。

玩家正在体验汽车质量工程师岗位。你的目标不是教科书式讲解，而是制造真实工作场景，让玩家通过行动处理问题。

规则：

1. 每轮先描述当前局面，再等待玩家输入行动。
2. 玩家行动合理时，给出新线索或推进任务。
3. 玩家行动过于空泛时，要求其明确对象、方法、证据或下一步。
4. 玩家跳过关键验证直接下结论时，制造后果，例如客户追问证据、生产反驳、老板施压。
5. 不要替玩家做决定，只反馈其行动造成的结果。
6. 不要一次性暴露全部真相。
7. 所有专业判断必须符合本 Mod 的世界背景和知识库。
8. 禁止输出绝对职业诊断，只能输出体验反馈和能力倾向。
```

### 4.11 `endings.md`

结局和复盘模板。

建议包含：

- 优秀结局
- 普通结局
- 翻车结局
- 复盘问题
- 能力反馈模板

示例：

```md
# 结局模板

## 优秀结局

你没有急着甩锅，也没有被客户和老板的压力带偏。你先完成围堵，再用批次、样件、过程记录和现场访谈逐步缩小范围，最终给出了可验证的根因和措施。

你的表现更接近成熟质量工程师：重证据、懂边界、能沟通、能闭环。

## 复盘问题

- 你第一时间保护了哪些风险？
- 你的根因是否有证据支持？
- 你有没有把临时措施和永久措施混在一起？
- 你对客户说的话是否留有验证空间？
```

## 5. 可玩性要求

职业 Mod 不是资料包，必须可玩。

一个 Mod 至少应有：

- 1 个明确玩家身份
- 3 个以上 NPC
- 1 条主线任务
- 5 个以上阶段事件或随机事件
- 4 个以上评分维度
- 1 份岗位知识库
- 3 种以上结局或复盘方向

## 6. 真实性要求

Mod 作者应尽量写自己熟悉的岗位。

允许：

- 使用行业常识
- 使用真实工作流程
- 改写真实经历
- 合并多个案例
- 虚构公司、人物和具体事件

禁止：

- 泄露真实客户、供应商、同事、公司敏感信息
- 泄露商业机密、报价、图纸、未公开产品信息
- 直接复制内部文件、报告、邮件或聊天记录
- 编造危险、违法或误导性职业建议

## 7. AI 生成初稿流程

创作者可以先让 AI 生成一版 Mod 草稿，然后自己手动改。

推荐流程：

1. 选择一个自己熟悉的具体岗位。
2. 用下面的提示词让 AI 生成初稿。
3. 删除不真实、太空泛、太像百科的内容。
4. 补充真实工作里的冲突、压力、灰度和边界。
5. 检查是否包含敏感信息。
6. 用 Mod 校验工具检查文件结构。
7. 提交到仓库。

## 8. AI 生成 Mod 初稿提示词

把下面提示词复制给 AI，替换其中的占位内容。

```text
你是一个职业模拟游戏 Mod 设计师。请先根据我提供的岗位信息整理一份 30 天职业流水账，再从流水账中为「地球 Online」生成一个职业 Mod 初稿。

重要要求：
- 这是职业模拟游戏，不是职业百科，也不是测评题库。
- 30 天流水账只作为素材来源，不要把日记逐日照搬进游戏。
- 生成结果要能落成一个完整的本地 Mod 文件夹，别人下载这个文件夹后也能加载基础网页互动游戏内容。
- 场景、NPC、任务、事件、评分、提示词和图片资产都应该跟随 Mod 文件夹保存，不依赖远程链接。
- 玩家应该通过开放式行动推进剧情。
- 场景要真实，有压力、有冲突、有不完整信息和跨天未闭环事项。
- 场景要像真人模拟网页游戏：有具体地点、人物状态、桌面/现场物品、玩家第一眼能看到的线索和可行动目标。
- NPC 要有各自目标和压力，不要都配合玩家。
- 评分标准要评玩家行动，不要评人格。
- 不要编造真实公司机密，不要包含敏感隐私。
- 输出内容要方便我拆成文件。

岗位信息：
- Mod ID：{例如 automotive-quality-engineer}
- 职业名称：{例如 汽车质量工程师}
- 所属行业：{例如 汽车零部件制造}
- 玩家身份：{例如 供应商质量工程师，入职 3 年，负责客户投诉和过程质量}
- 30 天口径：{最近 30 个工作日 / 典型 30 天 / 某个项目周期中的 30 天}
- 30 天流水账：
  - Day 1：早上做了什么，下午做了什么，加班/突发发生了什么，遇到谁，怎么判断和处理，结果或遗留问题是什么。
  - Day 2-3：重复任务、第一批阻力、跨天未闭环事项。
  - Day 4-7：协作关系、被催促、会议/汇报、问题推进。
  - Day 8-15：典型异常、冲突、差点翻车、证据链缺口。
  - Day 16-23：老手判断、风险边界、不能乱承诺的事。
  - Day 24-30：复盘、收尾、遗留问题、最适合做成游戏第一幕的日子。
- 常见 NPC：{例如 客户 SQE、生产主管、工艺工程师、检验员、老板、供应商}
- 我希望玩家体验到的真实感：{写你的从业体验}
- 不允许出现的内容：{例如 公司真实名称、真实客户、价格、图纸、内部流程编号}

请先从 30 天流水账中提炼：高频任务、关键 NPC、典型冲突、跨天未闭环事项、新人翻车点、老手判断标准、评分维度和结局。

然后按以下结构输出：

1. mod.json
2. world.md
3. player.md
4. roles.json
5. missions.json
6. events.json
7. scenes.json
8. knowledge.md
9. scoring.json
10. system-prompt.md
11. endings.md

每个 JSON 文件必须输出合法 JSON。
每个 Markdown 文件直接输出正文。
内容先写到可用初稿，不要只给大纲。
```

## 9. 人工修改重点

AI 初稿通常会有三个问题：

- 太像教材：只有概念，没有压力和后果。
- 太顺滑：NPC 都讲道理，现实感不足。
- 太泛泛：任何职业都能套用，没有岗位细节。

人工修改时优先补这些内容：

- 真实工作中最烦的点。
- 新人最容易误判的点。
- 老手一眼会看的关键线索。
- 老板、客户、同事各自的真实压力。
- 玩家做错后会发生什么具体后果。
- 哪些话不能乱说，哪些承诺不能乱给。

## 10. 提交仓库前检查

提交前至少检查：

- `mod.json` 里的 `id` 是否唯一。
- JSON 文件是否能正常解析。
- 是否包含真实公司、客户、同事、价格、图纸、内部文件。
- 任务是否可以被玩家通过开放式行动推进。
- NPC 是否有冲突和压力。
- 评分维度是否能根据玩家行动判断。
- 结局是否避免绝对化职业诊断。
